# AGENTS.md

## 适用范围

本文件适用于仓库根目录及所有子目录。若子目录另有更近的 `AGENTS.md`，以更近文件为准。

## 沟通方式

- 向用户汇报时使用中文，专有名词、命令、路径、代码标识符保留英文。
- 最终回复只写结论、实际改动、原因、验证结果。
- 最终回复不描述执行过程，不使用表演化措辞，不写无关技术细节。
- 多方案决策必须列出优缺点，并明确推荐方案和原因。

## 三个入口文档

| 文档 | 用途 |
|---|---|
| `README.md` | 项目总览、构建入口、文档导航 |
| `AGENTS.md` | AI 协作规则、开发硬约束、提交流程 |
| `PROGRESS.md` | 倒序进展日志、当前状态、剩余问题 |

规则：
- README 不写协作硬约束。
- AGENTS 不写流水账进度。
- PROGRESS 不写长期愿景。
- 新增文档前先判断能否合并进已有文档；能合并就不新建。

## 开发流程

每开始一个任务，先定义完成标准。交付前必须按完成标准验证。

代码或文档发生实质改动后：
1. 同步更新相关文档。
2. 运行与改动范围匹配的验证。
3. 更新 `PROGRESS.md` 顶部日志。
4. 提交并推送到 `origin/main`。

推送失败时，重试 2 到 3 次。仍失败时，在回复里明确说明本地 commit 未推送，不得写成已推送。

禁止：
- 不使用 `--no-verify`。
- 不使用 `--force`。
- 不推送非默认分支，除非用户明确要求。
- 不提交敏感文件、证书、账号、token、个人本地状态。

## Git 提交规则

commit message 使用中文，标题不超过 70 个字符。

标题动词使用固定词汇：
- 新增
- 修改
- 修复
- 删除
- 重构
- 文档
- 整理
- 清理
- 同步

正文需要包含：
- 改动原因
- 改动清单
- 验证结果
- 影响范围

## PROGRESS 规则

`PROGRESS.md` 使用倒序记录，最新条目放最上方。

每条记录包含：
- 日期和序号
- 任务
- 核心交付
- 验证结果
- 关键发现，可选
- 决策结论或下一步，可选
- commit hash
- push 状态

## 验证规则

凡是声称完成，必须有当前验证结果。

文档改动至少验证：
- 文件存在。
- 关键内容可读。
- `git diff --check` 无格式错误。

代码改动按影响范围验证：
- 能跑单测就跑单测。
- 能跑静态检查就跑静态检查。
- iOS 构建相关改动优先使用 Xcode workspace 和对应 scheme 验证。
- 验证入口不明确时，先查现有 README、workflow、fastlane 配置，再选择最小有效命令。

不得把没有执行过的验证写成已通过。

## 文档规则

- 文档正文使用中文。
- 路径、命令、代码标识符、框架名保留英文。
- 日志型内容倒序排列。
- 结构型内容按主题和使用顺序排列。
- 文档中的状态使用“已 / 待 / 未实现”，避免含糊的未来时态。
- 非平凡 bug、工具异常、构建坑需要记录到 `docs/工具与踩坑.md`。该文件不存在时，先创建再记录。

## 代码规则

- 同一事实、常量、配置、schema 只保留一个权威来源。
- 相似代码第 1 次和第 2 次可以直接写；第 3 次必须考虑抽 helper、模块或共享测试基建。
- 同一概念全项目使用同一名称。
- 优先复用既有模块，不复制整段代码改少量内容。
- 不为假设需求提前设计复杂抽象。
- 测试里的小重复可以接受；共享加载、临时目录、mock 工厂等基础能力应集中。
- 第二次手敲同一长命令链时，应做成脚本或统一命令。

## Sub-agent 使用规则

任务符合以下任一条件时，直接拆分给 sub-agent，无需询问用户：
- 可拆成多个平行且无依赖的子任务。
- 子任务职责明确分离，合并执行会污染上下文。
- 存在大量结构相同的重复任务。
- 子任务需要不同模型配置或权限策略。

## 本项目特殊规则

- 不提交 `LoopWorkspace.xcworkspace/xcuserdata/` 下的个人本地界面状态。
- 修改 iOS 构建、证书、Bundle ID、GitHub Actions、fastlane 配置前，先阅读相关现有说明文档。
- 涉及真实签名、发布、TestFlight、证书和账号时，只报告当前确认到的状态，不用推测替代结果。
- 本仓库包含多个嵌套子项目和 submodule；改动前确认目标文件属于当前仓库管理范围。
