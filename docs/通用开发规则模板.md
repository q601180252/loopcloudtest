# 开发规则模板
## §0 项目特定信息（每个新项目自己填）

```
项目名          : <name>
仓库地址        : <git url>
默认分支        : <master / main>
私有 / 公开    : <private / public>
主语言         : <Dart / Java / Go / ...>
构建工具       : <gradle / mvn / cargo / npm / ...>
测试入口       : <command>
静态检查入口   : <command>
设备白名单     : <若 mobile 项目>
包名固定       : <若 Android>
```

---

## §1 语言

- **正文一律 <主语言>**（中文 / 英文 / ...）。包括所有 `*.md`、git commit message、PR 描述、issue 评论、AI 助手交接。
- **代码标识符 / 命令 / 路径 / 框架专名照常英文**。
- 选定后**全项目一致**，不要部分中文部分英文。

> libre2hex 选中文。

## §2 三大入口文档（强制）

| 文档 | 单句概括 | 答的问题 |
|---|---|---|
| `README.md` | 总览 + 子项目索引 + 文档导航 + 快速开始 | "这项目是啥 / 怎么跑起来" |
| `<AI>.md`（如 `CLAUDE.md`） | AI 协作硬约束 + 常用命令 + 工作流约定 | "AI 接手要遵守什么" |
| `PROGRESS.md` | 倒序进展日志 + 状态总览 + 剩余问题 | "现在做到哪 / 下一步做啥" |
| `docs/目标.md`（可选） | 终极目标 + 当前边界 + 缺口 + 路线图 | "我们到底要做到什么程度" |

**绝对不混淆**：README 不写约束，CLAUDE 不写进度，PROGRESS 不写愿景。

## §3 目录布局

| 目录 | 内容 |
|---|---|
| 根 `/` | 顶层入口三大文档（命名固定）|
| `docs/` | 分支文档按主题切分；文件名可中可英但**单项目内统一** |
| `prompts/` | AI 助手交接模板（一 AI 一文件）|
| `tools/` | 一次性 / 辅助脚本 |
| `tests/` 或 `test/` | 自动化测试 |
| `examples/` 或 `example/` | 调用示例 |

**新加 `*.md` 前先想**：能不能塞进已有文档？能塞就**不新建文件**。

## §4 倒序约定（强制）

**日志型字段**（新条目放最上面）：

- `PROGRESS.md` 进展日志
- `CHANGELOG.md`（按 [Keep a Changelog](https://keepachangelog.com/) 风格）
- 已知坑 / FAQ
- 决策记录

**结构型字段**（按主题/顺序排，不倒序）：

- README 文档地图 / 子项目布局 / 快速开始
- 配置参考 / API 文档 / 状态总览表

## §5 同轮 commit + push（强制）

每完成"一件可独立交付的事"（不分大小）：

1. `git add -A`
2. `git commit -m "<语言一致的 message>"`
3. `git push origin <默认分支>`
4. **同轮**立刻同步 `PROGRESS.md` 顶部新日志条目（含 commit hash + 验收数字）

网络抖动失败 → 重试 2-3 次仍失败 → 在汇报里**明说**"本地 commit X 未 push"，不要假装推完了。

**禁**：`--no-verify` / `--force` / 推非默认分支 / 改 `.gitignore` 加敏感文件（除非用户明示授权）。

## §6 commit message 格式

```
<动词> <对象> <简短结果> （≤ 70 字符，标题）

<空行>

<正文：动机 / 改动清单 / 验收数字 / 影响范围>

<空行>

Co-Authored-By: <AI 名> <noreply@anthropic.com>   ← 若 AI 协作
```

**动词**用一致词汇表：新增 / 修改 / 修复 / 删除 / 重构 / 文档 / 整理 / 清理 / 同步。

## §7 字段时效性

- 文档与代码**同步更新**，改了代码就改对应文档段；不改不让 commit 走。
- "我们做到哪"答案只看 `PROGRESS.md` 顶条 + `docs/目标.md §当前边界`，其他地方写的"已完成"都是潜在过期。
- "AI 助手能不能跑某命令"答案只看 `<AI>.md` 硬约束 + `docs/工具与踩坑.md` 顶部最新坑。
- 撞到非平凡 bug / 工具异常：**立刻**追加 `docs/工具与踩坑.md` 顶部一条（症状 / 根因 / 解决 / 避免 / 参考五段），**不要等任务结束**。

## §8 验收命令约定

凡是声称"X 完成"必须能跑出一行明确 OK：

- 测试：`<test runner>` 末尾打印 `vectors OK <N>/<M>` 或 `tests passed: <N>/<N>`
- 静态检查：`<linter>` 打印 `No issues found!` / `0 errors`
- 数值验证用 bit-exact（IEEE-754 用 `floatToRawIntBits` / `byteArrayEquals`，不用 `==` / 容差）

所有验收命令汇总在 `README.md` 快速开始段，加新测试**同步加一行**+ 一行成功标准。

## §9 PROGRESS 日志条目模板

```
### <日期> <序号> — [一句话概括本里程碑]

- **任务**：本轮做什么 + 承接哪条上游
- **核心交付**：
  1. 文件路径 + 用途
  2. ...
- **验收数字**：跑 test 打印 N/M
- **关键发现**（可选）：算法 / 公式 / 反编译地址
- **决策结论 / 下阶段命题**（可选）
- **撞坑**（可选）：明说已加 docs/工具与踩坑.md 第几号
- **commit hash**：xxxxxxx
- **push 状态**：✅ origin/<分支> / ⚠️ 本地 commit X 未推送
```

## §10 prompts/ AI 交接模板

每个 AI 助手一个 prompt 文件，**self-contained**（必读文档清单 / 硬约束 / 工具栈 / 工作流 / 当次任务），让 AI 不需读项目历史就能开干。

约定：
- `prompts/<ai-name>-handoff.md`
- 内含 §A 开场白（每次复用）+ §B 当次任务（按 phase 选）
- §B 每个 phase 写清"输入素材 / 交付物 / 验收命令"
- 不假设上下文：必读 6 个核心文档清单 + 10 条硬约束

## §11 反向工程 / 调研产物

阶段性反编译 / 仿真 / 抓 dump / 调研结论放专门目录（如 `reverse_dumps/` `investigations/`）：

- 文件名英文 + 阶段编号（方便 grep）
- 内容用项目语言
- 与对应 dump bin / json 同目录
- PROGRESS 日志只**引用**这些 audit，**不复制**内容进 PROGRESS

## §12 不重复写代码（DRY + YAGNI 平衡）

> 本节所有例子刻意写成**领域无关**的占位形式（`Foo` / `bar` / 通用 API/DB/CLI 场景），不针对任何具体项目。换到 web / CLI / 库 / 移动 / 数据管线 都同样适用。

### 12.1 单一信息源（Single Source of Truth）

每个**事实 / 常量 / 配置 / schema**只存一份。

- ❌ 三个文件里硬编码 `MAX_RETRY = 5` / `API_BASE_URL = "https://api.example.com"`
- ✅ 一处 `config.yaml` 或 `Constants.X` + 其他 import
- ❌ 测试 fixture 数据手抄到 5 个 test 文件里
- ✅ 一个 `fixtures/` 目录或 `testdata.json` 共享
- ❌ 文档把"FAQ Q3"复制到 3 个不同 `*.md`
- ✅ 一处 FAQ，其他文档写"详见 FAQ Q3"
- ❌ 数据库表结构定义出现在 SQL migration + ORM model + API schema 三处，改字段要同步三遍
- ✅ 选一处作权威（通常是 migration），其他两处由它生成 / 引用

### 12.2 Rule of Three：写第 3 次再抽

第 1、2 次相似代码允许直接 copy-paste（避免过早抽象）。**写第 3 次时停下来抽** helper / 类 / 模块。

- 第 1 处写：`open file → read → parse JSON → close` 直写
- 第 2 处需要类似：原样 copy 过去
- 第 3 处又要：停，抽 `loadJsonFile(path) → T` 一处

YAGNI 反例（**不要**这样过早抽象）：

- ❌ 写 `interface IFooHandler` + 4 个 stub 实现，因为"以后可能加第五个"
- ✅ 直接写一个 `handleFoo()`，加第二个时再考虑分流
- ❌ `class GenericRunner<T extends Task>` 想给所有任务共用，但 5 个任务字段差很多
- ✅ 5 个 runner 各自直写，相似度 < 70% 就不强抽
- ❌ 加 `options Map<String, dynamic>` 给"未来可能的扩展用"，当前一个 key 都没用
- ✅ 等真需要时再加

### 12.3 命名一致

同一概念全项目用**同一名字** + 同一风格：

- ❌ `userId` / `uid` / `user_id` / `account_id` 混用指同一东西
- ✅ 选定 `userId`（或 `user_id`，按语言风格），全项目 grep replace 改齐
- ❌ `FooBar.java` `foo_bar.py` `fooBar.go` `foo-bar.ts` 命名风格不一
- ✅ 单语言内按语言惯例一致（Java/C# PascalCase 文件、Python/Rust snake_case 文件、TS/JS camelCase 或 kebab-case 选一种...）
- ❌ 同一概念在 API 叫 `customer`，DB 叫 `account`，代码叫 `user`
- ✅ 全栈选一个词

### 12.4 复用既有模块，不 fork-and-modify

需要稍改某个既有函数 → **加参数** / **加新方法** / **包一层**，不要复制整个函数改两行。

- ❌ `parseConfigV1()` `parseConfigV2()` 90% 相同代码
- ✅ `parseConfig(input, {bool strict = false, int schemaVersion = 1})`
- ❌ `sendEmailToUser()` + `sendEmailToAdmin()` 两个函数差一行
- ✅ `sendEmail(recipient, ...)`
- ❌ 复制整个 service class 改个 DB 表名做"另一个版本"
- ✅ 注入 table name 参数 / 用泛型 / 用 strategy

例外（版本切换 / A-B 测试 / 大版本迁移期）允许临时并存，但要在 commit message + 文档明示**并存原因** + **预计何时合并**。

### 12.5 测试不重复造基建

- ✅ 一处 `testkit` / `test_helpers` / `conftest.py` / `setup.ts` 提供 fixture 加载、断言、mock 工厂、temp dir
- ✅ 每个 test 文件只写"输入 → 期望"那两行不同的
- ❌ 每个 test 自己写一遍 `assertJsonEqual` / `loadFixture` / `mockHttpClient`
- ❌ 每个 test 自己 `tempDir = mkdtemp()` / `defer rm(tempDir)`
- ✅ 一处 fixture：`withTempDir(t, (dir) => ...)`

### 12.6 文档不重复内容

- ✅ 一处事实 + 其他地方**链接**它（`详见 FAQ #N` / `详见 ADR-007` / `详见 README §3`）
- ✅ 命令清单只在 `README.md` 一处，其他文档不复制
- ❌ 把"硬约束 H1-H10"复制到每个 `prompts/*-handoff.md` 里 → 改一条要改 N 处
- ❌ API endpoint 列表同时在 README / docs / 代码注释三处出现

例外：AI 助手 prompt 必须 self-contained 时，让 prompt 模板**引用**核心文档路径，AI 第一步就 `cat` 它们，而不是把内容塞进 prompt。

### 12.7 配置 / 常量 / 阈值集中 + 注明来源

- ✅ 一个 `config.yaml` / `Constants.X` / `defaults.go` / `env.ts` 放所有可调常量
- ❌ 同一个魔术数字（`0.85` / `429` / `3600`）散落代码各处
- 注释带上**来源**让未来读者知道为什么是这个值：
  ```python
  RETRY_DELAY_MS = 250       # AWS SDK 推荐起点 (https://docs.aws.amazon.com/...)
  MAX_CONNECTIONS = 100      # 后端 nginx upstream max_conns
  TOKEN_EXPIRY_SEC = 3600    # 与 IdP 约定一致, 见 ADR-012
  ```

### 12.8 重复操作做成命令

第二次手敲同一系列命令时，做成 `Makefile` / `justfile` / `package.json scripts` / `.sh` 一键跑：

- ❌ 每次都敲 `cd web && npm run lint && npm test && cd ../api && go test ./... && cd ..`
- ✅ `make verify` / `just verify` 一条
- ❌ 部署步骤"先 build 再 push image 再 update k8s manifest 再 rollout"靠记忆
- ✅ `make deploy` 一条 + README 写"部署只准用 make deploy"

### 12.9 schema / 接口契约不重复定义

同一 schema 出现在前后端 / 多服务时：

- ✅ 选一个权威源（OpenAPI YAML / Proto / JSON Schema / Pydantic / dataclass），其他端**生成**而非手写
- ❌ 前端手抄一份 TS interface + 后端手抄一份 Go struct + DB 手抄一份 SQL，三处都要改

工具栈例：openapi-generator / buf / quicktype / json-schema-to-typescript / sqlc / Prisma。

### 12.10 反例：什么时候**应该**写重复

DRY 不是宗教。下列场合允许甚至鼓励重复：

- **跨边界**（micro-service / 公开 API / 不同部署单元之间）不要为了共享 schema 引入强耦合的共享库 → 让两边各自定义、用 contract test 保证一致
- **测试**（test 内的小重复比测试间相互引用更可读，每个 test 自己 setup 自己 fixture 是 OK 的）
- **相似但语义不同**：两段代码看起来一样但描述的是**不同业务概念**（如"订单金额计算" vs "退款金额计算"），强抽会把未来的差异变化压抑掉
- **DSL / 配置文件**（K8s manifest / Terraform / GitHub Actions YAML 重复是常态，硬抽反而不直观）

## §13 不要做的事（汇总）

文档侧：
- ❌ 新 `*.md` 不预想"能不能塞进已有文档" → 文档碎片化
- ❌ 倒序日志倒着倒着改成正序 → 阅读"最新"要翻到底
- ❌ 文档时态用"将要""未来" → 用"已"/"待"/"未实现"
- ❌ 验收命令没明确 OK 行 → 没法机器判定"完成"
- ❌ commit 后忘 push → 队列堆积，新会话误判状态
- ❌ 阶段日志没 commit hash → 历史无法回查
- ❌ 改硬约束不在文档里同步 → 下次协作者违反
- ❌ 文档 emoji / 装饰滥用 → 朴素优先

代码侧：
- ❌ 三处硬编码同一常量 → 改一处忘其他两处
- ❌ 复制整段代码改两行 → fork-and-modify 病
- ❌ 为假设需求过早抽象 → 抽象成本 > 复用收益
- ❌ 同概念多名 / 跨语言风格混 → 协作者认知负担
- ❌ schema 多处手抄 → 两边漂移
- ❌ 第二次手敲长命令链 → 该做成 make/just 还在敲历史

## §14 适配指引

把本模板拷到新项目时：

1. 填好 §0 项目特定信息
2. §1 选定语言后**全项目搜替**
3. §2 三大入口文档若已有就保留命名，没有就新建空骨架
4. §3 目录布局按项目实际删减
5. §5 默认分支按项目实际改（master / main）
6. §6 commit 动词词汇表按语言习惯调
7. §8 验收命令按 test runner 替换
8. §10 若不用 AI 协作可整段删
9. §11 若不是反向工程项目可删
10. §12 全部保留（DRY+YAGNI 是通用工程原则）
11. 加一条 §15 项目自己的特殊约定（如 libre2hex 的"设备白名单"）
